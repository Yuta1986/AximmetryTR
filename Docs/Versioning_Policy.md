# Versioning Policy

## 目的

この資料群では、`Git の履歴` と `配布版の版番号` を別物として扱います。

- Git: 途中の作業や細かな差分も含めて、すべて記録する
- 版番号: 配布物として見たときに、読者にとって明確な変更があったかを示す

## 基本ルール

- 読者に見える変更があった場合は、版番号を更新する
- 内部メモだけの更新、運用メモだけの更新、エクスポート用ツールだけの更新で、配布ガイド本文に影響しない場合は、版番号を更新しない
- Git のコミットは毎回行ってよいが、版番号更新は配布物の変更時だけにする

## 版番号の考え方

このプロジェクトでは `major.minor.patch` 形式で扱います。

### Major

`x.0.0` 相当。対象読者、資料の役割、大きな構成方針が変わるときに更新します。

例:

- 講師向け資料からクライアント向け資料へ作り替える
- 復習資料から完全手順書へ役割を変える
- 全体構成や章立てを大きく組み替える

### Minor

`0.x.0` 相当。章追加、対象範囲の拡張、明確な内容追加など、読者にとってまとまった変更があるときに更新します。

例:

- 新しい章を追加する
- トラブルシューティングの対象範囲を広げる
- 新しいワークフローや運用前提を追加する

### Patch

`0.0.x` 相当。構成は維持しつつ、読者に見える改善が入ったときに更新します。

例:

- 用語整理
- 説明文の明確化
- 注釈追加
- 誤記修正
- PDF / HTML の見た目改善
- 参照リンクやタイトル表示の改善

## この資料での運用ルール

- ルート直下には常に最新の配布版だけを置く
- 旧版の Markdown は `Achive/guide_versions/` に移す
- 旧版の生成物は `Achive/exports/` に移す
- 現行版の生成物は `build/exports/` に置く

## 版番号を上げるタイミング

次の流れで更新します。

1. 現行ガイドの内容を確定する
2. 変更の重さに応じて `major / minor / patch` を判断する
3. ファイル名と本文内の版番号を更新する
4. HTML / PDF を再生成する
5. 旧版のソースと生成物を `Achive` に移す
6. Git にコミットする

## 現在の判断基準

今回の一連の更新は、構成を保ったまま内容整理、用語明確化、注釈追加、エクスポート表示改善を行ったものなので、`patch` 更新として扱います。
